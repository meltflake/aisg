# i18n 规范（多语言开发与验收清单）

> 本规范定义 sgai 项目支持中英双语的方式。新增页面、新增数据、改动共享组件，全部按本规范执行；提交前必须通过 `node scripts/i18n-check.mjs` 验证。
>
> Last updated: 2026-05-02

## 1. 站点基本架构

- 默认语言 `zh`，路径不带前缀（`/policies/`、`/debates/budget-2914/` 等）。
- 第二语言 `en`，路径加 `/en/` 前缀（`/en/policies/`、`/en/debates/budget-2914/`）。
- 每个 `src/pages/<route>/<file>.astro` 都有 `src/pages/en/<route>/<file>.astro` 镜像页。Markdown 博客同时使用 `src/data/post/<slug>.md`（zh）与 `src/data/post/en/<slug>.md`（en），通过 frontmatter `lang: 'en'` 区分。
- 所有 i18n 工具函数集中在 `src/i18n/index.ts`：
  - `getLangFromPath(pathname)` → `'zh' | 'en'`
  - `localePrefix(lang)` → `'' | '/en'`
  - `localizedHref(path, lang)` → 给绝对路径加 lang 前缀
  - `unprefixed(path)` → 去掉 `/en/` 前缀
  - `pickLocalized(record, zhKey, enKey, lang)` → 任何"双字段"数据的统一取值入口
  - `channelLabel(channel, lang)` → SocialChannel 的显示文本
  - `t(lang, key)` → 字典查询（zh 字典是真值，EN 缺失时回落到 zh）

## 2. 数据契约（双字段约定）

> 一句话：数据文件以中文为主语言，英文以可选 `*En` 兄弟字段并列存储；展示由页面 + `pickLocalized` 决定。

### 2.1 命名规则

- 中文字段用裸名：`title`、`description`、`summary`、`role`、`statement`...
- 英文字段加 `En` 后缀：`titleEn`、`descriptionEn`、`summaryEn`、`roleEn`、`statementEn`...
- 数组字段同样加后缀：`keyDebates: string[]` ↔ `keyDebatesEn?: string[]`。
- TypeScript 接口里 `*En` 字段都标 `?` 可选；类型注释加一句"English sibling of X"。

```typescript
export interface KeyInsight {
  title: string;
  titleEn?: string;
  description: string;
  descriptionEn?: string;
  evidence: string;
  evidenceEn?: string;
  significance: 'high' | 'medium' | 'low';
}
```

### 2.2 例外：标题已是英文的字段

某些数据 `title` 字段本身存的是英文（例如国会辩论 Hansard 原始标题）。此时 `zhTitle` 是中文兄弟字段。这种"英文为主+中文翻译"的反向布局允许存在，但所有页面渲染时**仍然走 `pickLocalized`**，不要假设方向。

### 2.3 状态枚举值

`status: '已落实' | '进行中' | '仍在讨论'` 这种枚举型字符串，比较用原值（中文），但**展示给用户的字符串要在页面层做翻译映射**。具体可参考 `src/pages/en/debates/index.astro` 的 `statusLabels`。

如果将来要彻底干净，把枚举值换成 `'implemented' | 'in-progress' | 'under-discussion'`，再用页面层翻译。当前为兼容历史数据保持中文枚举值。

### 2.4 SocialChannel `label` / `labelEn`

- `label` 默认存 zh（如 `"官网"`、`"MDDI 官网"`、`"@joteo_ylm"`）。
- 若 `label` 含 CJK 字符，**必须**同时给 `labelEn`（英文等价物，如 `"Website"`、`"MDDI website"`）。
- 平台名（twitter、linkedin、facebook 等）由 `channelLabel()` 内置兜底，不需要逐条写。
- 例外：纯拉丁文标签（`@joteo_ylm`、`Newsletter`）不需要写 `labelEn`，渲染时直接复用。

## 3. 页面渲染规则

### 3.1 顶层契约

- 每个 `src/pages/en/...` 页面顶部声明 `const lang = 'en' as const;` 并把 `lang={lang}` 传给 `Layout`。
- 每个 zh 页面如果使用了任何接受 `lang` prop 的组件，也要传 `lang`（默认从路径推断，但显式传更稳）。
- 渲染数据字段时，**永远不要**直接 `{record.title}`。统一用：
  - `pickLocalized(record, 'title', 'titleEn', lang)` 或
  - `record.titleEn || record.title`（仅当 EN sibling 几乎总存在时简写）。

### 3.2 链接

- 跳到内部页面时，永远过一层 `localizedHref(path, lang)`。
- 不要硬编码 `/en/` 前缀；不要硬编码 `/people/...` 这类裸路径。
- 例：`<a href={localizedHref(`/people/${p.id}/`, lang)}>...</a>`

### 3.3 共享组件（zh + EN 都使用）

放在 `src/components/` 下、不分 lang 的组件（Header、Footer、Logo、LangBanner、LanguageToggle、Breadcrumb、AuthorBio、TableOfContents、NextPrevPost、RelatedRail、blog/ListItem、blog/SinglePost），必须**自己**从 URL 推断 lang：

```astro
---
import { getLangFromPath } from '~/i18n';
const lang = getLangFromPath(new URL(Astro.url).pathname);
const isEn = lang === 'en';
---
```

然后所有展示文案、内部链接、字段读取都按 lang 走。

### 3.4 子页面 zh 副标题（zhTitle, zhName, zhEvent）

- 在 zh 页面：英文条目可保留 zh 翻译做副显示，符合中文阅读体验。
- 在 EN 页面：**默认隐藏** zh 副字段。`personName` 卡片下方的 `zhName` 是少数允许的例外（短姓名做注音，已在 i18n-check 的 allow-list 里）。
- 一旦在 EN 页面渲染了某个 zh 副字段，i18n-check 会报错，强制做选择。

## 4. SEO 与 Metadata

- `<html lang>`：zh 页面输出 `lang="zh-CN"`；EN 页面输出 `lang="en"`。已由 `src/layouts/Layout.astro` 统一处理。
- `og:locale`：zh `zh_CN`、EN `en_US`。由 `src/components/common/Metadata.astro` 根据 lang 输出。
- `og:site_name`：zh `新加坡 AI 观察`、EN `Singapore AI Observatory`。来源是 `src/i18n/index.ts` 字典。
- `<title>` 模板：zh `%s — 新加坡 AI 观察`、EN `%s — Singapore AI Observatory`。
- `description` 兜底：每个页面应自带 EN description；缺失时回落 `t('en', 'siteDescription')`。
- `<link rel="alternate" hreflang>`：zh 页面输出 `hreflang="en"` 指向 `/en/...` 镜像，EN 页面输出 `hreflang="zh-CN"` 指向 zh 镜像。（已在 `CommonMeta.astro` 配置；新增页面如果走 PageLayout 自动继承，无需重复配置。）

## 5. 验证

### 5.1 自动检查（每次构建）

```bash
npm run build
node scripts/i18n-check.mjs
```

脚本扫 `dist/en/**/*.html`，剥离 `<script>` `<style>` `<head>` 与所有 HTML 属性，搜索 CJK 字符。命中即报错并定位到具体页面。

允许例外（已写在 `ALLOW_PATTERNS` 里）：
- `中文版可用` / `阅读中文版` / `中`：LangBanner 与 LanguageToggle 主动邀请用户切换到 zh 的副本。
- 短姓名（≤ 8 字 CJK）：作为 `zhName` 注音副显示存在（如 person/MP 卡片）。

如果你为某条特殊残留申请例外，**必须**在 `ALLOW_PATTERNS` 里加注释说明原因。

### 5.2 反向检查（zh 页面是否被英文污染）

目前没有自动化的"zh 页面英文残留"检查，因为 zh 页面允许大量英文（人名、机构缩写、产品名）。如果未来需要，可以加一个 `zh-check.mjs` 反向版，但允许列表会很长，价值有限。手工 review 即可。

### 5.3 PR / 提交前必跑

```bash
npm run check          # astro check + eslint + prettier + verify-graph
npm run build          # 必须先构建才能跑 i18n-check
node scripts/i18n-check.mjs   # 中文残留扫描
```

## 6. 新增内容时的清单

### 6.1 新增数据条目（debate / policy / lever / video / 等）

- [ ] zh 字段填全
- [ ] 每个用户可见的 zh 字段都加了 `*En` 兄弟字段（标题、摘要、要点、状态描述、机构名）
- [ ] 数组字段同样加了 `*En` 数组（即使内容相同也要复制一份，防止页面在 EN 下显示 zh）
- [ ] 跑一次 `npm run build && node scripts/i18n-check.mjs` 看新条目是否引入新残留

### 6.2 新增页面

- [ ] `src/pages/<route>/...astro`（zh）和 `src/pages/en/<route>/...astro`（EN）双份
- [ ] EN 页面顶部 `const lang = 'en' as const;` 并传给 Layout
- [ ] 渲染数据走 `pickLocalized` 或 `*En` 兜底
- [ ] 内部链接用 `localizedHref`
- [ ] 跑 `i18n-check` 通过

### 6.3 改动共享组件

- [ ] 用 `getLangFromPath(new URL(Astro.url).pathname)` 推断 lang
- [ ] 显示文案用 `t(lang, key)` 或本地三元
- [ ] 字段取值用 `pickLocalized`
- [ ] 链接用 `localizedHref`
- [ ] zh 与 EN 各跑一遍构建确认两端都对

### 6.4 SEO 调优

- [ ] 修改 `<title>` / description / og 标签时同时改 zh 与 EN 两份字典
- [ ] hreflang 必须同时存在（zh ↔ en 双向）
- [ ] 页面 metadata 不要硬编码"新加坡 AI 观察"或"Singapore AI Observatory"，统一走 `t(lang, 'siteName')`

## 7. 已知限制

- `src/data/post/en/*.md` 的 EN 博客以独立文件存在，与 zh 版手工保持一致；改 zh 文章不会自动同步到 EN，需要双改。
- `LangBanner` 在 EN 页面显示中文邀请文案（`中文版可用`、`阅读中文版`），是有意为之——这是给具有中文浏览偏好的访客看的。i18n-check 已为此加了白名单。
- Hansard 国会辩论的"speakers"字段保持英文姓名（与 Hansard 原文一致）。
